# Clarify: Phase 4 Collaborative & Vault Lifecycle (v0.9.x)

This document initiates Phase 1 (Clarification) of the spec-driven development process for **Phase 4: Collaborative & Vault Lifecycle** of Project Mythrax, covering:
1.  **4.1 Git-Backed Federated Memory (Team Synapse)**
2.  **4.2 Biological Decay & Cognitive Sleep (Active Forgetting)**
3.  **4.3 Self-Healing Memory Calibration & Citations**

---

## Restated Request

Implement team-wide memory sharing, database biological decay, self-healing calibrations, and citation footnote tracking:
*   **Git-Backed Federated Memory**: Support a dual-vault structure: the **Local Developer Vault** (`[vault_root]/`) and a **Project Team Vault** (`<project_root>/.mythrax-shared/`) committed to Git.
    *   *Promotion*: If a local rule's utility score reaches $\ge 50.0$, automatically copy it to the shared vault under `.mythrax-shared/wisdom/proposed/` and notify the developer.
    *   *Conflict Resolution*: Implement a merge command/hook that scans pulled changes, detects duplicate/conflicting rules, and automatically merges them or writes a `conflict.md` for manual resolution.
    *   *Sanitizer*: Create a git pre-commit hook that runs the `SecretFilter` on all files in `.mythrax-shared/` to strip credentials before committing.
*   **Biological Decay & Cognitive Sleep**:
    *   *Decay Loop*: On every retrieval cycle, decay the utility scores of unselected nodes using an exponential decay formula:
        $$U_{new} = U_{old} \times e^{-\lambda \Delta t}$$
        where $\Delta t$ is the time since the last retrieval/creation and $\lambda$ is the decay rate.
    *   *Sleep Compaction & Archiving*: During daily compaction, identify nodes with utility below a threshold (e.g. $5.0$). Strip their vector embeddings, move their markdown files to `vault/archive/`, and delete their active SurrealDB records to save memory while keeping them readable in Obsidian.
*   **Self-Healing Memory Calibration & Citations**:
    *   *Auditor Daemon*: Periodically generate synthetic queries from vault nodes, run vector searches, and automatically optimize HNSW parameters or similarity thresholds if the target node is missed.
    *   *Citations & Footnotes*: Track every memory node loaded and cited during a session. Automatically append a markdown footnote block listing clickable links to the exact `WisdomRules` and `Episodes` that influenced the task plan or git commit message.

## Known Facts

### 1. Codebase Architecture
*   **SecretFilter**: Defined in `mythrax-core/src/secret_filter.rs` and contains regex patterns to redact credentials.
*   **Wisdom & Episode Schemas**: Defined in `mythrax-core/src/contracts.rs`.
*   **Vector Search & Indexing**: Handled by `SurrealBackend` in `mythrax-core/src/db/backend.rs` using HNSW index parameters.
*   **CLI Subcommands**: Handled in `mythrax-core/src/cli.rs` and `main.rs`.

### 2. File Watchers & Stores
*   **MarkdownStore**: Manages all file creations and directory initializations under the vault root.
*   **Watcher**: Monitors file events recursively and syncs changes to SurrealDB.

## Assumptions

1.  **Utility Score Tracking**: We can add a `last_retrieved_at` timestamp field to `WisdomRule` and `Episode` schemas (mapping to database columns) to calculate the time delta $\Delta t$ for exponential decay.
2.  **Shared Vault Path**: The project team vault is located at `<project_root>/.mythrax-shared/`. We can resolve `<project_root>` by reading the `MYTHRAX_WORKSPACE_ROOT` environment variable.
3.  **Active Citation Tracking**: The Mythrax MCP server can maintain a list of cited node IDs in the active session's Short-Term Memory (STM), and append them as a formatted footnote to git commit messages and task lists generated by the agent.
4.  **HNSW Calibration**: We can adjust the HNSW cosine similarity threshold dynamically in `SurrealBackend` by updating a configuration table in SurrealDB, ensuring search sensitivity adapts to the auditor's findings.

---

## Ambiguities

1.  **Selection (Citation) Detection**: How does the database know which search results were actually "selected" or "used" by the agent to trigger utility reinforcement instead of decay?
    *   *Resolution*: When the agent completes a turn and saves an episode representing its work, it lists the node IDs it cited (or the server parses the citations from the agent's task plan). Alternatively, we expose a simple MCP tool `cite_memory(node_ids: Vec<string>)` that the agent calls, or the server automatically reinforces any rules whose `target_pattern` matched during the turn. We will implement both: automatic reinforcement for matched rules, and an MCP tool for explicit citations.
2.  **Conflict Adjudication Criteria**: When two shared rules conflict, how does the adjudicator resolve them?
    *   *Resolution*: The adjudicator compares the `utility` scores and `timestamp` fields. The rule with the higher utility score wins. If utility scores are equal, the newer rule (by timestamp) wins. The losing rule is moved to `.mythrax-shared/wisdom/conflict_archive/` and a `conflict.md` warning is generated.

---

## Tradeoffs

*   **Continuous Decay Database Writes**: Updating utility scores on every search query incurs database write overhead.
    *   *Remedy*: To optimize performance, we will batch decay calculations. Instead of writing on every single query, we perform the decay calculation dynamically *during retrieval* (adjusting the returned score on the fly) and write the updated utility scores back to the database in a single batched background write at the end of the session.
*   **LSP vs Regex AST for Citations**: Extracting code symbols to match citations exactly is complex. We will leverage our lightweight regex-based symbol extractor from Phase 3, keeping the citation engine extremely fast and simple.
