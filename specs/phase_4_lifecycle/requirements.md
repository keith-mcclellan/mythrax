# Requirements: Phase 4 Collaborative & Vault Lifecycle (v0.9.x)

This document defines the requirements for Phase 4 of the Mythrax 0.9.x features.

---

## Problem

1.  **Siloed Learning**: Developers and agents working in separate environments have no way to share high-utility wisdom rules. When one developer solves a critical database lock or compiler bug, other team members do not benefit from this learning, leading to repeated trial-and-error mistakes across the team.
2.  **Long-Term Database Pollution**: Over months of development, the memory database accumulates hundreds of old, one-off episodes and obsolete wisdom rules. This bloats the active vector search space, degrades retrieval performance, and increases latency.
3.  **Static Retrieval Thresholds**: Vector similarity thresholds are static and hardcoded. As the database grows, static thresholds lead to either false negatives (missing critical rules) or false positives (bloating the prompt with irrelevant context).
4.  **Invisible Cognitive Influence**: There is no audit trail or transparency showing which past episodes, experiences, or wisdom rules influenced the agent's code changes, making it difficult for developers to verify the safety and rationale of the agent's decisions.

---

## Outcome

*   **Federated Team Synapse**: High-utility rules are automatically promoted to a shared project vault committed to Git. Conflicting rules from merges are automatically adjudicated, and pre-commit hooks guarantee that no credentials ever leak to GitHub.
*   **Active Forgetting (Decay & Sleep)**: Unused memories exponentially decay over time. Obsolete, low-utility memories are put to "cognitive sleep"—their embeddings are stripped, their database records deleted, and their markdown files archived, saving memory while keeping them readable in Obsidian.
*   **Self-Healing & Clickable Citations**: An auditor daemon periodically runs synthetic searches to dynamically calibrate similarity thresholds. Every task plan and git commit generated by the agent automatically includes clickable footnotes citing the exact source memories that influenced the changes.

---

## User Value

*   **Multiplier Team Learning**: One developer's fix instantly becomes a shared rule that protects the entire team from making the same mistake.
*   **Sub-5ms Search Latency Forever**: Active forgetting keeps the database slim, ensuring search speeds and token costs remain low over years of use.
*   **Complete Audit Transparency**: Total visibility into the agent's reasoning. Clickable footnotes link every code change directly to its historical source rule or episode in Obsidian.

---

## In Scope

*   **Git-Backed Federated Memory (Team Synapse)**:
    *   Support the **Local Developer Vault** (`[vault_root]/`) and the **Project Team Vault** (`<project_root>/.mythrax-shared/`) resolved via `MYTHRAX_WORKSPACE_ROOT`.
    *   **Promotion**: Auto-copy rules with utility $\ge 50.0$ to `.mythrax-shared/wisdom/proposed/`.
    *   **Conflict Resolver**: Implement `mythrax merge-vault` to resolve conflicting rules based on utility and timestamp, archiving losers and creating `conflict.md` for ties.
    *   **Sanitizer**: Create a pre-commit hook that runs `SecretFilter::clean` on all files in `.mythrax-shared/`.
*   **Biological Decay & Cognitive Sleep (Active Forgetting)**:
    *   **Decay Loop**: Add a `last_retrieved_at` timestamp field to `WisdomRule` and `Episode` schemas. On every search retrieval, calculate and apply exponential decay:
        $$U_{new} = U_{old} \times e^{-\lambda \Delta t}$$
        to all unselected nodes, and reinforce selected/cited nodes (resetting utility or incrementing it). Batch write updates at the end of the session.
    *   **Sleep Compaction & Archiving**: During daily compaction, identify nodes with utility $< 5.0$. Strip their embeddings, move their files to `vault/archive/`, and delete their active SurrealDB records.
*   **Self-Healing & Footnote Citations**:
    *   **Auditor Daemon**: Periodically generate synthetic queries, run searches, and dynamically adjust similarity thresholds or HNSW parameters in SurrealDB to optimize retrieval.
    *   **Citations**: Track cited node IDs in the session STM and automatically append a clickable markdown footnotes block to task plans and git commit messages.

---

## Out of Scope

*   Managing remote git push, pull, or merge conflicts directly (handled by standard developer git commands).
*   Fine-tuning or training embedding models locally.

---

## Inputs

*   **For Federated Memory**: Files under `.mythrax-shared/`, local utility scores, and git merge/commit status.
*   **For Decay**: Retrieval queries, selection/citation events, and timestamp deltas.
*   **For Auditor & Citations**: Vault nodes, synthetic query results, and agent-generated task lists or git commits.

---

## Outputs

*   **For Federated Memory**: Promoted rule files in `.mythrax-shared/wisdom/proposed/` and sanitized, conflict-free shared vaults.
*   **For Decay & Sleep**: Decayed utility scores, archived files under `vault/archive/`, and a slim active SurrealDB index.
*   **For Auditor & Citations**: Dynamically calibrated similarity thresholds in SurrealDB, and formatted markdown citations footnotes.

---

## Constraints

*   **Pre-Commit Speed**: The pre-commit sanitizer must run in $<500\text{ms}$ to avoid slowing down git commits.
*   **Decay Batch Performance**: Re-calculating and batch-writing decayed utility scores must run completely asynchronously at the end of a session, adding $0\text{ms}$ latency to active search queries.
*   **Footnote Validity**: Clickable footnote links must use the correct local absolute file URI format to ensure they open directly in Obsidian and the editor.

---

## Risks and Edge Cases

*   **Accidental Redaction of Safe Data**: The pre-commit sanitizer might redact strings that look like secrets but are actually safe.
    *   *Remedy*: The `SecretFilter` patterns are highly targeted to assignments (e.g. `api_key = "..."`) to minimize false positives, and the developer can bypass the hook using `--no-verify` if needed.
*   **Decay of Pinned Rules**: Pinned rules could accidentally decay.
    *   *Remedy*: Rules with `tier = "pinned"` are explicitly excluded from the biological decay loop, retaining their maximum utility score permanently.

---

## Acceptance Criteria

*   **[ ] AC-4.1 (Rule Promotion)**: Saving a local rule with utility $\ge 50.0$ automatically copies the rule markdown file to `<project_root>/.mythrax-shared/wisdom/proposed/`.
*   **[ ] AC-4.2 (Conflict Resolution & Sanitizer)**: Running `mythrax merge-vault` successfully resolves conflicting rules (moving the lower-utility rule to `conflict_archive/`), and the pre-commit sanitizer successfully redacts API keys in `.mythrax-shared/` before committing.
*   **[ ] AC-4.3 (Biological Decay)**: Performing multiple searches where a rule is unselected successfully decays its utility score exponentially in SurrealDB, while selected rules are reinforced and have their `last_retrieved_at` updated.
*   **[ ] AC-4.4 (Cognitive Sleep Archiving)**: During daily compaction, any rule or episode with utility $< 5.0$ has its SurrealDB record deleted, its embedding stripped, and its markdown file moved to `vault/archive/`.
*   **[ ] AC-4.5 (Auditor Calibration & Citations)**: The auditor daemon successfully runs a synthetic search and optimizes the HNSW similarity threshold. Generating a git commit or task list automatically appends a clickable footnotes block listing the exact cited memory paths.
